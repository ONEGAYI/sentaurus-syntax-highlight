# 函数文档提取与编写规范

本文档定义了项目中 JSON 函数文档的结构规范，确保所有语言的文档格式一致、质量可控。当前已覆盖以下文档：

| 文档文件 | 语言/工具 | 条目数 | 文件位置 |
|----------|----------|--------|---------|
| SDE 函数文档 | SDE (Scheme) | ~565 | `syntaxes/sde_function_docs.json` |
| Scheme 内置函数文档 | SDE (Scheme) | 191 | `syntaxes/scheme_function_docs.json` |
| SDevice 命令文档 | SDevice (Tcl) | ~2123 | `syntaxes/sdevice_command_docs.json` |
| Svisual 命令文档 | Svisual (Tcl) | ~257 | `syntaxes/svisual_command_docs.json` |
| Inspect 命令文档 | Inspect (Tcl) | ~190 | `syntaxes/inspect_command_docs.json` |
| SProcess 命令文档 | SProcess (Tcl) | ~164 | `syntaxes/sprocess_command_docs.json` |
| Tcl 核心命令文档 | 所有 Tcl 语言共用 | 66 | `syntaxes/tcl_command_docs.json` |
| Tcl 数学函数文档 | 所有 Tcl 语言共用 | 31 | `syntaxes/tcl_expr_mathfunc_docs.json` |
| Tcl 子命令文档 | 所有 Tcl 语言共用 | 5 主命令 | `syntaxes/tcl_subcommand_docs.json` |

## 1. JSON 结构总览

### 1.1 通用字段（所有文档必备）

```jsonc
{
  "<key>": {
    "signature": "<函数签名>",       // 必备
    "description": "<功能描述>",     // 必备
    "parameters": [                  // 必备，可为空数组
      {
        "name": "<参数名>",
        "type": "<参数类型>",
        "desc": "<参数说明>"
      }
    ],
    "example": "<使用示例>"          // 必备
  }
}
```

### 1.2 扩展字段（按语言需要）

| 字段 | 适用语言 | 说明 |
|------|---------|------|
| `section` | SDevice, Inspect, Svisual | 命令所属的功能分类。SDevice 为顶层 section 名；Inspect/Svisual 为功能分组名 |
| `keywords` | SDevice, Inspect, Svisual | 辅助模糊搜索的关键词列表（2-5 个） |

### 1.3 常量特殊字段（PhysicalConstants 等）

```jsonc
{
  "const::ElementaryCharge": {
    "section": "PhysicalConstants",
    "signature": "$const::ElementaryCharge",
    "description": "Elementary charge constant (1.602×10⁻¹⁹ C).",
    "parameters": [],
    "example": "set q $const::ElementaryCharge",
    "keywords": ["charge", "physical", "constant"]
    // 无额外 value 字段 — 常量值直接嵌入 description
  }
}
```

## 2. 条目键（Key）命名规范

| 语言 | Key 格式 | 示例 |
|------|---------|------|
| SDE | `模块名:函数名`（kebab-case，冒号分隔） | `sdegeo:create-circle`、`sdedr:define-constant-profile` |
| Scheme 内置函数 | 纯函数名 | `caaar`、`string-append`、`list` |
| SDevice | PascalCase 命令名 | `Electrode`、`Physics`、`CurrentPlot` |
| Inspect | 纯命令名（命令原样，含下划线） | `cv_create`、`proj_load`、`f_VT`、`const::ElementaryCharge` |
| Svisual | 纯命令名（命令原样，含下划线） | `load_file`、`create_plot`、`get_dataset` |
| Tcl 共享 | 纯命令名 | `set`、`foreach`、`string` |

> **规则**：Key 必须与 `all_keywords.json` 中的关键词完全一致，不得自造名称。

## 3. 各字段编写规范

### 3.1 signature（函数签名）

描述函数/命令的调用语法，使用纯文本表示。

| 语言 | 格式 | 示例 |
|------|------|------|
| SDE | `(模块:函数名 参数1 参数2 [可选参数])` | `(sdegeo:create-circle center-pos radius region-material region-name [start-angle end-angle])` |
| Scheme | `(函数名 参数1 参数2)` | `(car pair)`、`(string-append string1 string2)` |
| SDevice | `命令名 { 参数1=<类型> 参数2=<类型> }` | `Electrode { Name=<string> Voltage=<float> }` |
| Inspect/Svisual | `命令名 <参数1> <参数2> ?可选参数?` | `cv_create <name> <xExpr> <yExpr> ?y2?` |
| Tcl 共享命令 | `命令名 <参数1> <参数2> ?可选参数?` | `string match ?-nocase? <pattern> <string>` |

**规则**：
- 可选参数用方括号 `[]` 包裹（SDE/Scheme/SDevice）或 `? ?` 包裹（Inspect/Svisual/Tcl）
- SDevice 的参数赋值使用 `=` 语法
- 保持签名在一行内，过长时可省略不常用的可选参数
- Inspect/Svisual 的参数使用 `<name>` 表示必选、`?name?` 表示可选

### 3.2 description（功能描述）

用简洁的英文描述函数/命令的用途和行为。

**规则**：
- 首句直接说明功能（"Creates..."、"Specifies..."、"Returns..."）
- 不超过 2-3 句话
- 如有重要的使用前提或约束，在描述中提及
- 不使用 Markdown 格式

### 3.3 parameters（参数列表）

每个参数为包含 `name`、`type`、`desc` 三个字段的 JSON 对象。

#### name（参数名）

- SDE/Scheme：使用 kebab-case，如 `center-pos`、`region-material`
- SDevice：使用 PascalCase，如 `Name`、`Voltage`、`Method`
- Inspect/Svisual/Tcl：使用尖括号包裹的描述性名称，如 `<name>`、`<xExpr>`、`<curve>`
- 占位符参数用尖括号包裹，如 `<vector>`、`<plot_variables>`

#### type（参数类型）

使用以下标准类型标签：

| 类型标签 | 含义 | 适用场景 |
|---------|------|---------|
| `string` | 字符串 | 材料名、文件名、正则表达式 |
| `number` | 数值（实数或整数） | 通用数值参数 |
| `int` | 整数 | 迭代次数、索引 |
| `float` | 浮点数 | 电压、坐标、容差 |
| `position` | 位置/坐标 | `(position x y z)` 形式的坐标 |
| `pair` | Scheme 对 | `car`/`cdr` 操作的参数 |
| `list` | Scheme 列表 | 列表操作函数的参数 |
| `boolean` | 布尔值 | 开关选项 |
| `curve` | 曲线引用 | Inspect 公式函数中的曲线引用，如 `<curveName>` |
| `enum` | 枚举值 | 固定取值集合的选项 |

#### desc（参数说明）

**规则**：
- 首句说明参数是什么（可附带 Sentaurus 类型标注，如 `(REAL)`、`(POSITION)`）
- 可选参数必须注明 "Optional"
- 如有默认值或约束条件（如 "must be greater than zero"），在说明中包含
- **参数选项必须解释实际行为和含义**，不能只是枚举可选值。信息来源优先级：命令附录 → 正文相关章节 → AI 推断
- 枚举值格式：列出每个选项时用括号标注其含义，如 `"solid" (continuous line), \"dash\" (dashed line), \"dot\" (dotted line)`

**示例对比**：

```jsonc
// SDE 风格 — 选项带行为解释
{ "name": "face_pattern", "type": "string", "desc": "Optional. Specifies the face pattern for 3D face contacts. Can be one of: \"solid\" (filled face), \"###\" (crosshatch), \"||\" (vertical lines), \"==\" (horizontal lines), \"//\" (diagonal lines). Default is \"##\"." }

// SDevice 风格 — 行为描述
{ "name": "Voltage", "type": "float", "desc": "Contact voltage in V. Can also use (val1 At val2, ...) for transient piecewise-linear voltage." }

// SVisual 风格 — 从手册获取行为含义
{ "name": "line_style", "type": "string", "desc": "Style of the line. Possible values: \"solid\" (continuous line), \"dash\" (dashed line), \"dot\" (dotted line), \"dashdot\" (alternating dash and dot), \"dashdotdot\" (alternating dash and two dots)." }

// 错误示范 — 只枚举值不解释
{ "name": "line_style", "type": "string", "desc": "solid, dash, dot, dashdot, dashdotdot" }
```

### 3.4 example（使用示例）

提供一个最小可用的调用示例。

**规则**：
- SDE/Scheme 示例可直接求值（Scheme 示例可用 `=>` 标注返回值）
- SDevice 示例展示完整的 Tcl 块语法（含花括号和缩进）
- 多行示例使用 `\n` 换行
- 优先使用具体的数值和名称，避免过于抽象

**示例**：

```jsonc
// SDE — 单行
"example": "(sdegeo:create-circle (position 0 0 0) 10 \"Silicon\" \"Region_1\")"

// Scheme — 含返回值
"example": "(cadr '(a b c))  => b"

// SDevice — 多行
"example": "Electrode {\n  Name=\"gate\"\n  Voltage=0.0\n  Material=\"Aluminum\"\n}"

// Inspect — 真实用例
"example": "cv_create j($n) \"$desFile anode OuterVoltage\" \"$desFile cathode TotalCurrent\""

// Inspect 公式函数 — 在 cv_compute 中使用
"example": "cv_compute \"vecmax(<P($n)>)\" A A A A"

// Svisual — 带可选参数
"example": "load_file /path/to/Structure.tdr -geoms {0 2}"
```

### 3.5 section（所属模块）

标注该命令的**主** section（默认/最常用的上下文）。

**已知 section 值**：

| 语言 | section 值 |
|------|-----------|
| SDevice | `File`、`Device`、`Electrode`、`System`、`Solve`、`Physics`、`Math`、`Plot`、`Contact`、`Mesh` 等 |
| Inspect | `General`、`File I/O`、`Curves`、`Attributes`、`Data Access`、`Transform`、`Computing`、`Extraction`、`Extraction Library`、`extend Library`、`Formula`、`PhysicalConstants`、`IC-CAP`、`Curve Comparison`、`Script Control` |
| Svisual | `File`、`Dataset`、`Plot`、`Curve`、`Contour`、`Cutline`、`Annotation`、`Tool` 等 |

### 3.6 contexts（多 section 上下文语义，仅 SDevice）

当同一个关键词在不同父 section 下有**不同语义**时，使用 `contexts` 字段记录每个 section 特定的描述。

**结构**：
```jsonc
{
  "Current": {
    "section": "Electrode",        // 主 section
    "description": "...",          // 默认描述（向后兼容）
    "contexts": {                  // 可选，仅在多语义时添加
      "Electrode": "Current boundary condition in A/um/d. Voltage is used as initial guess only.",
      "Plot": "Specifies which current-related quantities to write to the current plot file.",
      "Solve": "Solve for the current equation.",
      "Goal": "Use current value as the solve target."
    }
  }
}
```

**规则**：
- 仅在 AI 验证确认有多 section 语义差异时才添加 `contexts`
- `contexts` 的 key 是父 section 名称（PascalCase）
- `contexts` 的 value 是该上下文下的语义描述字符串
- 若一个词在所有 section 中语义一致，不需要 `contexts` 字段
- 主 `description` 保持不变（向后兼容），`contexts` 是补充信息

### 3.7 keywords（子关键词列表，仅 SDevice）

列出该命令/section 下所有合法的子关键词（KEYWORD2/KEYWORD3 级别）。

**规则**：
- 仅列出直接子关键词，不递归展开嵌套层级
- 如无子关键词，使用空数组 `[]`
- 关键词名称使用 PascalCase

## 4. 质量检查清单

编写或提取文档时，逐条确认：

- [ ] **Key 唯一性**：JSON 中不存在重复 key
- [ ] **Key 一致性**：Key 与 `all_keywords.json` 中的关键词匹配
- [ ] **字段完整性**：必备字段（`signature`、`description`、`parameters`、`example`）均不为空
- [ ] **参数类型**：所有 `type` 字段使用标准类型标签
- [ ] **可选参数**：`signature` 中用 `[]` 标注，`desc` 中注明 "Optional"
- [ ] **示例有效性**：示例语法正确、可直接使用
- [ ] **JSON 合法性**：文件为合法 JSON（无尾逗号、无注释）
- [ ] **SDevice 附加**：`section` 和 `keywords` 字段均已填写
- [ ] **多语义覆盖**：若关键词在多个 section 中有不同语义，`contexts` 字段已填写且每条描述准确反映该上下文下的含义（仅 SDevice）
- [ ] **keywords/parameters 完整性**：对照官方手册确认 `keywords[]` 和 `parameters[]` 无遗漏
- [ ] **中英文条目数一致**：英文和中文 JSON 文件的 key 数量完全相同
- [ ] **示例真实性**：Inspect/Svisual 的 example 优先从 `display_test/` 范本或手册中选取真实用例，而非编造
- [ ] **共享文档去重**：Tcl 方言的专有文档不应包含已被 `tcl_command_docs` / `tcl_expr_mathfunc_docs` / `tcl_subcommand_docs` 覆盖的命令

## 5. 国际化（i18n）

每个英文文档对应一个中文翻译文档，命名约定为 `<name>_docs.zh-CN.json`。

**规则**：
- JSON 结构（key、字段名）与英文版完全一致
- 仅翻译 `description`、`parameters[].desc` 的文本内容
- `signature`、`parameters[].name`、`parameters[].type`、`keywords[]`、`section` 不翻译
- `example` 不翻译（保持代码原样）
- 技术术语保持英文原文（如 "Newton solver"、"Schottky contact"）
- 使用 `docs/glossary.json` 作为术语统一参考
- SDevice 的 `contexts` 值翻译为中文

## 6. 新语言文档的创建流程

当需要为新的 Sentaurus 工具创建函数文档时：

1. **收集关键词源**：从 `syntaxes/<module>.tmLanguage.json` 和 `syntaxes/all_keywords.json` 中提取关键词列表
2. **确定格式模板**：根据语言类型选择对应的文档模板
   - Scheme 方言 → 参照 `sde_function_docs.json` 格式（7.1 节模板）
   - Tcl 方言（section 嵌套型）→ 参照 `sdevice_command_docs.json` 格式（7.2 节模板）
   - Tcl 方言（命令扁平型）→ 参照 `svisual_command_docs.json` 或 `inspect_command_docs.json` 格式（7.4 节模板）
3. **编写英文文档**：按第 3 节规范逐条编写
4. **质量检查**：按第 4 节清单逐条确认
5. **编写中文文档**：按第 5 节规范翻译
6. **集成到扩展**：在 `src/register-completion-providers.js` 的 `langSpecificDocs` IIFE 中添加新语言的分支

## 7. 附录：文档模板

### 7.1 Scheme 方言模板（SDE 风格）

```jsonc
{
  "<module>:<function-name>": {
    "signature": "(<module>:<function-name> <param1> <param2> [<optional-param>])",
    "description": "One or two sentences describing what this function does and any important behavior notes.",
    "parameters": [
      {
        "name": "<param1>",
        "type": "<type>",
        "desc": "Description of param1 (SENTAURUS_TYPE). Additional constraints or defaults."
      },
      {
        "name": "<optional-param>",
        "type": "<type>",
        "desc": "Optional. Description of optional-param."
      }
    ],
    "example": "(<module>:<function-name> <value1> <value2> \"material\" \"region\")"
  }
}
```

### 7.2 Tcl 方言模板（SDevice 风格）

```jsonc
{
  "<CommandName>": {
    "section": "<SectionName>",
    "signature": "<CommandName> { <param1>=<type> <param2>=<type> }",
    "description": "One or two sentences describing what this command does and any important behavior notes.",
    "parameters": [
      {
        "name": "<ParamName>",
        "type": "<type>",
        "desc": "Description of ParamName in V/A/etc. Constraints or format notes."
      }
    ],
    "example": "<CommandName> {\n  <ParamName>=<value>\n  <ParamName2>=<value>\n}",
    "keywords": [
      "<SubKeyword1>",
      "<SubKeyword2>"
    ]
  }
}
```

### 7.3 Tcl 命令扁平型模板（Inspect / Svisual 风格）

适用于命令为顶层 proc 调用、无嵌套 section 结构的 Tcl 方言工具。

```jsonc
{
  "command_name": {
    "section": "Functional Group",
    "signature": "command_name <param1> <param2> ?optionalParam?",
    "description": "One or two sentences describing what this command does.",
    "parameters": [
      {
        "name": "<param1>",
        "type": "string",
        "desc": "Description of param1."
      },
      {
        "name": "?optionalParam?",
        "type": "string",
        "desc": "Optional. Description of optionalParam. Default behavior when omitted."
      }
    ],
    "example": "command_name arg1 arg2",
    "keywords": ["related", "words", "for", "search"]
  }
}
```

**子代理文档提取工作规范（通用，适用于所有语言）**：

1. **信息来源优先级**：官方手册正文 > 手册附录命令参考 > 范本文件中的真实调用 > AI 推断（需注明）
2. **example 优先使用真实用例**：从 `display_test/` 范本或手册示例中选取，而非编造。如果手册和范本中都没有该命令的示例，可以使用 AI 构造的简单示例
3. **共享文档去重**：Tcl 方言的专有文档**不得**包含已被以下共享文档覆盖的命令：
   - `tcl_command_docs.json`（Tcl 核心命令：set, puts, if, for, foreach, proc 等）
   - `tcl_expr_mathfunc_docs.json`（数学函数：sin, cos, sqrt, abs, log 等）
   - `tcl_subcommand_docs.json`（子命令：string match, file exists 等）
4. **同名命令覆盖**：如果语言专有库中的命令与核心命令同名但语义增强（如 extend 库的 `cv_write`），专有文档会覆盖共享文档（合并顺序决定），这种情况允许保留
5. **section 值必须从规范中选取**：每个语言的 section 值在 3.5 节中列出，不得自造
6. **parameters 必须完整**：对照手册的语法描述，确保所有参数均被记录。标志类参数（如 `-flag`）的 type 为 `boolean`
7. **批量提取时保持一致性**：同一批次内的文档，description 风格、参数命名、示例格式应保持一致

### 7.4 SDevice 审计产出模板（子代理参照）

子代理在审计/补全流程中，对每个关键词产出以下格式的文档片段：

```jsonc
{
  "<KeywordName>": {
    "section": "<PrimarySection>",
    "signature": "<KeywordName> { <param>=<type> ... }",
    "description": "功能描述（基于官方手册，简洁准确）",
    "parameters": [
      { "name": "<Param>", "type": "<type>", "desc": "参数说明" }
    ],
    "keywords": ["<SubKeyword1>", "<SubKeyword2>"],
    "example": "<KeywordName> {\n  <Param>=<value>\n}",
    "contexts": {
      // 仅在 AI 验证确认有多 section 语义差异时填写
      "<SectionA>": "该关键词在 SectionA 上下文中的具体语义",
      "<SectionB>": "该关键词在 SectionB 上下文中的具体语义"
    },
    "_audit": {
      "isNew": false,
      "keywordsComplete": true,
      "parametersComplete": true,
      "missingKeywords": [],
      "sourceHeadings": ["Math", "Controlling Transient Simulations"],
      "notes": ""
    }
  }
}
```

**子代理工作规范**：

1. **信息来源优先级**：官方手册正文 > 手册附录命令参考 > AI 推断（需在 `_audit.notes` 中注明）
2. **contexts 判断**：必须基于手册中该词在不同上下文的实际描述，不能仅凭关键词列表推断
3. **keywords/parameters 完整性**：对照手册中该 section 的完整语法描述，补全遗漏的子关键词和参数
4. **新增条目**：对当前 `sdevice_command_docs.json` 中不存在的词，`_audit.isNew` 设为 `true`，所有字段需完整填写
5. **已有条目补全**：仅补全缺失字段，不覆盖已有内容（除非已有内容明显错误）
6. **`_audit` 字段**：仅用于审计合并阶段的质量追踪，最终合并到产物文档时移除
